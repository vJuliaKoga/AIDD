## 4. 技術的リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| Deep Eval（G4）の精度不足 | High | Medium | Promptfooで継続評価。閾値を段階的に調整 |
| OpenAI API コスト超過 | Medium | Low | G4呼び出しログを保存し月次で確認。月額1万円を上限に設定 |
| SQLiteのパフォーマンス限界 | Low | Low | 1人開発のため問題なし。将来PostgreSQL移行を検討 |
| React Flow の学習コスト | Medium | Medium | 公式サンプルから始め、最小構成で実装する |
| Playwright のWindows動作 | Medium | Low | Playwright公式Win32サポートあり。Edge/Chrome両方でテスト |

## 5. 実装の原則

1. **AIDDで開発する** — 要件定義YAMLを先に作成し、品質ゲートを通過させてから実装に入る
2. **品質ゲートを先に実装する** — ゲート（G1〜G5）を最初に動かし、以降の開発でも常に使う
3. **テストを書きながら実装する** — 機能実装→即テスト追加。テストなしでの Done 禁止
4. **docs/ を常に最新に保つ** — 実装が仕様から逸脱した場合はどちらかを修正する
5. **CLAUDE.md / progress.txt を毎セッション更新する** — AIが前回の状態を把握できるようにする
