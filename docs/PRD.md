# PRD — 上流工程品質保証の構造化・可視化プラットフォーム

> source: planning/planning_v2.2.md (PLN-PLN-GOAL-001)
> version: v2.2 / 2026-02-21

---

## 1. プロダクト概要

### 1.1 一言定義

AI駆動開発（AIDD）における上流工程の品質保証を **構造化・可視化・検証可能** にするWebアプリケーション。

### 1.2 位置づけ

本プロダクトは **サンプルプロジェクト**として設計されており、AIDDの品質保証手法そのものを実証することが一次目的である。このアプリ自体をAIDDで開発することで、提示する手法の実効性を証明する。

| 側面 | 説明 |
|------|------|
| 一次目的 | QA4AIDDの実践手法を確立し、実証する |
| 二次目的 | 現場への段階的導入の足掛かりとする |
| 実証方法 | このアプリ自体をAIDDで開発し、品質保証プロセスを適用 |
| 成果物 | 動作するアプリ + CI/CD環境 + 品質保証テンプレート集 |

## 2. 解決する問題

### 2.1 現場課題

| カテゴリ | 具体的課題 | 影響 |
|----------|-----------|------|
| 仕様の曖昧性 | 「適切に」「柔軟に」等の曖昧語が頻出 | AIが誤解釈し意図しない実装を生成 |
| 仕様と実装の乖離 | 要件定義・設計書・実UIが一致しない | テスト工数が計画比2.5倍増加（仮説） |
| QA観点の属人化 | 何を確認すべきか個人の経験に依存 | 品質のバラつき、新人の戦力化遅延 |
| 判断根拠の不在 | なぜその設計にしたか記録されない | 保守時の意図不明、同じ議論の繰り返し |
| 品質の不可視性 | QA実施有無は分かるが中身が見えない | 品質リスクの早期検知不能 |

### 2.2 根本原因

1. **構造化されていない知識** — QA観点がドキュメントに散在
2. **検証不可能な記述** — テスト設計ができない要件定義
3. **トレーサビリティの欠如** — 要件と実装の紐付けがない

## 3. ゴールと成功指標

### 3.1 Primary Goal

仕様書の妥当性を検証可能にし、仕様と実装の乖離を防止する

### 3.2 Secondary Goals

- 上流工程のQA観点を標準化・構造化する
- 判断根拠とプロセスを記録・可視化する
- AIDDにおける品質保証手法を確立・実証する
- 検索工数の削減とPoC段階での観点提案を可能にする

### 3.3 KPI（Phase別）

| Phase | 成功指標 | 測定方法 |
|-------|---------|---------|
| Phase 0 | アプリ完成 + CI/CD構築 | デプロイ成功 |
| Phase 1 | 仕様書起因のバグ30%削減 | バグチケット分類 |
| Phase 2 | 手戻り工数20%削減 | 工数ログ比較 |
| Phase 3 | トレーサビリティ100%達成 | 自動チェックスクリプト |
| Phase 4 | プロンプトDB 100件以上 | DBレコード数 |

## 4. ユーザーと制約

### 4.1 ターゲットユーザー

- AI駆動開発を実践するエンジニア・QAエンジニア
- **必須スキル**: JSON・YAMLの構造を読み書きできること

### 4.2 技術制約

| 制約項目 | 内容 |
|---------|------|
| AI基盤 | OpenAI API のみ使用可能 |
| OS環境 | Windows のみ |
| 開発体制 | 1人での開発・検証 |
| 対応ブラウザ | Edge、Chrome（モバイル非対応） |

### 4.3 プロジェクト制約

| 制約項目 | 内容 |
|---------|------|
| 予算 | 月額10万円以内 |
| 期間 | 明確な期限設定なし（品質優先） |
| スコープ | サンプルプロジェクトとしての完成度を優先 |

---

## 5. スコープ

### 5.1 スコープ内

- 上流工程（要件定義・基本設計・詳細設計）の品質保証ワークフロー
- 構造化テンプレート・チェックリスト・プロンプト集の提供
- 品質ゲート評価の自動化（Deep Eval、JSON Schema検証）
- トレーサビリティID体系の実装
- CI/CD環境構築とAllure Reportによるモニタリング
- Promptfooによるプロンプト品質評価
- Abort理由記録と判断ログの保存
- JSON/YAMLエクスポート機能
- データベースによる承認者管理

### 5.2 スコープ外（将来拡張候補）

| 項目 | 将来優先度 |
|------|-----------|
| アプリ内AI自動生成機能 | 低 |
| 外部ツールAPI連携 | 中 |
| モバイル対応 | 低 |
| プロンプト共有機能 | **最高** |

## 6. 機能要件

### 6.1 コア機能

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| F-001 | 品質ゲート実行 | G1〜G5の品質ゲートを自動評価し合否を返す |
| F-002 | タスクチケット管理 | L1（工程）/L2（タスク）/L3（詳細）の3層構造でチケットを管理 |
| F-003 | ステータス管理 | ToDo/InProgress/Done/Abort の遷移管理 |
| F-004 | Abort記録 | カテゴリ選択+50文字以上の理由記述を必須化 |
| F-005 | 判断ログ記録 | decidedby/decidedat/decisionsummary を必須化 |
| F-006 | トレーサビリティ管理 | derived_from/traces_to による要件→設計→テストの紐付け |
| F-007 | 曖昧語リアルタイム検出 | テキスト入力時に曖昧語を検出してアラート表示 |
| F-008 | エクスポート | JSON/YAML/PDF形式での出力 |
| F-009 | スナップショット保存 | プロジェクト状態の時点保存 |
| F-010 | 品質スコア推移グラフ | Deep EvalスコアのAllure Reportへの連携 |

### 6.2 品質ゲート定義

| ゲート | 評価内容 | 合格基準 | 自動検証 |
|--------|---------|---------|---------|
| G1: 曖昧語チェック | 曖昧な表現の有無 | 曖昧語0件 | 正規表現 |
| G2: 検証可能性 | 受入基準の存在と具体性 | 全要件に検証可能な基準 | チェックリスト |
| G3: 構造一貫性 | JSON Schema準拠 | スキーマ検証100%パス | JSON Schema Validator |
| G4: AI評価スコア | Deep Evalによる評価 | スコア0.8以上 | Deep Eval |
| G5: トレーサビリティ | ID紐付けの完全性 | 全要件が設計・テストに紐付く | カスタムスクリプト |

## 8. 設計哲学

「考えなくていいQA」ではなく「考える順番を固定するQA」

| 役割 | 担当 | 理由 |
|------|------|------|
| 判断 | 人間 | 責任の所在を明確にする |
| 記憶 | システム | 観点の漏れを防ぐ |
| 構造化 | システム | 思考順序を標準化する |
| 検証 | 自動化 | 機械的チェックは自動化 |
