## 1. 設計原則

### 1.1 コンポーネント設計の基本方針

1. **3クリック原則を守る** — 工程選択 → タスク選択 → 詳細確認で完結する導線を常に意識する
2. **段階的情報開示（Progressive Disclosure）** — 初心者モードと熟練者モードで表示情報量を切り替える
3. **状態は Zustand に集約する** — コンポーネントローカルの状態は最小限にとどめる
4. **型安全を徹底する** — `any` 禁止。API レスポンスは Zod スキーマで必ず検証する
5. **UIの統一性を担保する** — 同じ用途のUI要素には同じコンポーネントを使う（タスクカードは必ず `TaskCard` を使うなど）

### 1.2 品質目標

- UI コンポーネントのテストカバレッジ: 80%以上（Storybook + Vitest）
- Playwright E2Eテストでの主要ユーザーフロー検証: 全SCR対象
- Visual Regression Testing（Chromatic）でのUI統一性確認

## 5. 曖昧語リアルタイム検出

```typescript
// lib/ambiguous-words.ts

export const AMBIGUOUS_WORDS = [
  '適切に', '十分に', '柔軟に', '簡単に', '最適に',
  '基本的に', '一般的に', '必要に応じて', '適宜', 'なるべく',
  'できるだけ', 'なるべく早く', 'スムーズに', 'うまく',
] as const;

export function detectAmbiguousWords(text: string): string[] {
  return AMBIGUOUS_WORDS.filter(word => text.includes(word));
}

// hooks/useAmbiguousWordCheck.ts
export function useAmbiguousWordCheck(value: string) {
  const detected = useMemo(() => detectAmbiguousWords(value), [value]);
  return { detected, hasIssues: detected.length > 0 };
}
```

## 6. APIクライアント

```typescript
// lib/api.ts — すべてのAPI呼び出しはこれを経由する
const BASE_URL = 'http://localhost:3001/api';

async function apiRequest<T>(
  endpoint: string,
  options?: RequestInit,
  schema?: z.ZodSchema<T>
): Promise<T> {
  const res = await fetch(`${BASE_URL}${endpoint}`, {
    headers: { 'Content-Type': 'application/json' },
    ...options,
  });
  if (!res.ok) throw new ApiError(res.status, await res.json());
  const data = await res.json();
  return schema ? schema.parse(data) : data; // Zod検証
}
```

## 7. モード切替（初心者 / 熟練者）

| 機能 | 初心者モード | 熟練者モード |

## 9. コーディング規約

- **ファイル名**: PascalCase（コンポーネント）/ camelCase（hooks, lib, types）
- **エクスポート**: named export のみ（default export 禁止）
- **型**: `interface` を優先（`type` は Union型のみ）
- **`any` 禁止**: 代わりに `unknown` + 型ガードを使う
- **`console.log` 禁止**: デバッグ目的でも残さない
- **コメント**: 自明でないロジックのみ記述（コードが語るものにはコメント不要）
- **Tailwind**: クラス名の順序は `prettier-plugin-tailwindcss` に従う
