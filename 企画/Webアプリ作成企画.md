# 上流工程品質保証の構造化・可視化プラットフォーム

# 1. 企画背景

AI駆動開発（AIDD）の導入により、
要件定義・設計工程においてAIの利用が増加している。

しかし、現場では次の課題がある。

### ■ 課題

- 上流工程のQA観点が属人化している
- チェックリストやプロンプトが散在している
- 「何を確認したか」が記録されない
- Abort判断や差し戻し理由が残らない
- 進捗と品質保証範囲が可視化されない

結果として、

> QAの実施有無は分かるが、
> QAの“中身”が見えない

状態になっている。

---

# 2. 本企画の目的

本企画は、上流工程の品質保証を

- 構造化する
- 可視化する
- 再利用可能にする
- 判断ログを残す

ためのワークフロー基盤を構築するものである。

---

# 3. 基本思想

### 「考えなくていいQA」ではなく「考える順番を固定するQA」

本アプリはAIに判断させる仕組みではない。

- プロンプトはテンプレートとして保管
- チェックリストは標準観点として整理
- 判断は必ず人が行う

目的は

> QAの思考順序を標準化すること

である。

---

# 4. コンセプト

## 押せば展開されるQAワークフロー

工程スイッチを押すと、

- 必要なタスクが展開
- タスクを開くと観点・テンプレート表示
- 実施ステータスを記録
- フロー図として可視化

する構造。

---

# 5. システム構造

## A. 階層構造

| レイヤー | 内容                                     |
| -------- | ---------------------------------------- |
| L1       | 工程（要件定義／基本設計／詳細設計）     |
| L2       | タスク（非機能確認、前提整理など）       |
| L3       | 詳細（チェック観点・テンプレ・参考資料） |

---

## B. タスクチケット構成

各タスクには以下を保持する：

- 目的
- 確認観点
- チェックリスト
- 参考プロンプト（テンプレ）
- 参考資料リンク
- 判定基準

※自動生成は行わない

---

# 6. ステータス設計

各タスクに以下を持つ：

- ToDo
- Done
- Abort（理由必須）

### Abort設計思想

Abortは例外ではない。

- 「今回対象外」
- 「前提未確定」
- 「スコープ外」

などの理由をログ化する。

Abortも正式な品質判断として扱う。

---

# 7. 可視化機能

## ■ 品質フロー図生成

- 実施済タスクを接続
- Abort分岐も表示
- プロジェクト固有の品質履歴を生成

---

## ■ スナップショット保存

- セッション保存
- JSON形式エクスポート
- 日報利用可能

---

# 8. UI設計における品質観点

本企画ではUI統一性も検証対象とする。

### ■ 統一対象

- 工程スイッチUIの統一
- タスクチケットUIの統一
- 親子UIの差別化

これは

> 設計仕様に対する再現性の確認

であり、プロンプトの品質検証にも活用可能。

---

# 9. 本基盤の位置づけ

本基盤は

- AI評価エンジンではない
- 自動判定システムではない
- AI制御プラットフォームではない

あくまで

> 上流QAを構造化する土台

である。

---

# 10. 実行ロードマップ

| Phase | 内容                 |
| ----- | -------------------- |
| 1     | 要件定義工程のみ実装 |
| 2     | 設計工程追加         |
| 3     | ログ分析機能         |
| 4     | 全社展開             |

---

# 11. 期待効果

- QA観点の標準化
- 判断ログの蓄積
- 属人性の低減
- 上流工程の見える化
- 精神的負担の軽減

---

# 12. 結論

本企画は、
AIを制御するシステムではなく、
AIを使う時代においても、人が責任を持てるQAの仕組みを整えるための基盤
である。

品質保証を「個人の力量」から
「構造とログ」に移す第一歩とする。
